# Redis 자료 구조 활용 사례
> 레디스의 자료구조를 활용하면 복잡한 어플리케이션의 연산을 자료구조의 특징으로 대체할 수 있다.

### 리더보드 (Sorted Set)
1. 월별 리더보드, 주간 리더보드, 일별 리더보드의 구현이 가능하다.
2. 20250506의 일별 리더보드는 다음과 같이 구성한다.
    1. daily_board:20250506을 키로 한다.
    2. value로는 key : value = user_id : user_score로 한다.
    3. 이렇게 저장하면 value 기준으로 자동 sort가 된다.
3. 주간 리더보드의 합산을 주기적으로 수행한다면 ?
    1. ZUNIONSTORE 연산을 통해서 수행이 가능하다.

#### 추가 고려사항
1. 주간 리더보드의 합산을 매주 1회 갱신이 아니라, 실시간으로 제공해야 한다면..?
    1. 주간리더보드를 하나 만들고 그 리더보드에 비동기로 업데이트를 치는 방식
    2. 주간리더보드의 결과를 캐싱을 해두고,, 캐시를 주기적으로 업데이트한다.
        1. 캐시 업데이트의 주기마다 worker redis에서 읽기 연산을 돌려서 캐시를 최신화한다.
    3. 구조의 개선을 통해서 가능한 부분도 있다.
        1. 누적 점수를 매일 계산한다. 계산된 누적 점수는 하루동안은 유지된다.
        2. TOP-N 만을 유지하고 누적 처리를 한다.
        3. 샤딩하여 논리적으로 계산한다.
            1. 유저별로 샤드를 나누어서 다른 키의 zset에 저장하고, TOP-10만 뽑아서 어플리케이션에서 논리적으로 계산한다.
            2. TOP-N을 유지한다면 아주 효율적인 방식이다.
        4. 비동기로 갱신하며 worker가 stream을 받아서 주간보드를 갱신한다.
2. 클러스터 구성을 통해서 트래픽 해소가 가능할까 ?
    1. primary-replica 설정을 통해서 해결이 가능하다.
    2. 무거운 읽기 연산용 replica / 사용자 읽기 연산용 replica 구성
    3. 실시간 데이터 + 쓰기용으로 primary를 사용

### 최근검색기록 (Sorted Set)
1. 유저별 최근 검색 기록을 저장한다.
2. user_id 별로 하나의 zset을 관리하며, score는 시간기준으로 한다.
3. 검색하면 ZINCRBY를 사용하던지, ZADD를 활용해서 갱신한다.
4. 읽기전에 인덱스를 활용해서 최대 갯수를 이외의 검색어는 제거한다.

### post-tag 기능 (Set)
1. post에는 tag를 달 수 있다.
2. post키에 tag를 데이터로 넣을 수도 있다.
3. tag키에 post의 번호를 데이터를 넣는다.
4. 각 set의 inter 연산을 통해서 원하는 tag를 동시에 갖고 있는 post를 뽑는다던가의 연산이 가능하다.

#### 고려사항
1. 너무 많은 post가 존재한다면..? post의 번호 기준으로 샤딩하여 저장한다.

### 랜덤 추출
1. hash, set, zset 등 각 키들중 랜덤값을 추출할 수 있다.

### 좋아요
1. 댓글의 id 기준으로 set을 구성하고, 좋아요를 누른 유저의 id를 set의 member로 추가한다.
2. 반대로 유저 기준으로 set을 구성하고 유저 set에 좋아요를 누른 댓글의 번호를 저장한다.
3. 이때 둘다 샤딩을 하여 분산 저장한다.

### 채널의 안읽은 메시지 수 카운팅
1. 카카오톡 채널당 안읽은 메시지의 수를 카운팅해야한다.
2. 유저당 하나의 hash를 갖고 이 hash의 key는 채널방,  value는 읽지 않은 메시지의 갯수를 저장한다.
3. 인메모리 redis에 저장하고 유저가 읽으면 value의 값을 감소시킨다.

#### 추가 고려사항
1. 채널당 속해있는 유저는 서버의 DB로 관리된다.
2. 어떤 유저가 글을 보내고 글을 지운다면, 채널방에 속해있는 유저의 목록을 갖고 소켓 통신을 하고 있는 단말 어플리케이션 프로세스에 카운터 다운 통신을 시도한다.

### DAU (bitmap)
1. bitmap은 string의 최대 길이인 512MB의 데이터를 저장할 수 있다.
2. bit로 계산하면 거의 1000만개의 bit이다.
3. 따라서 1000만 유저의 체크로직이 가능하다. (유저의 id가 0이상의 정수 단위여야한다.)
4. dau는 실시간일 필요는 없으니 비동기 구조로 bitmap의 bit를 세팅한다.
5. 해당 일자의 bitmap의 bitcount를 활용하여 방문자 수를 구할 수 있다.
6. 누가 방문했는지는 응용단에서 계산이 가능하다. bit -> int 연산으로 가능하다.
7. 출석 이벤트의 경우도 서로 다른 일자의 bitmap의 공통된 비트를 구하여 offset 계산을 하면 된다.

### 미터링 서비스 (Hyperloglog)
미터링 서비스는 사용자가 얼마나 많이 서비스를 사용했는지를 측정해야한다.   
예를 들어서 로그를 보내고 집계하여 통계를 제공하는 서비스가 있고 이 서비스는 사용자가 얼만큼의 로그를 누적했는지를 토대로 과금한다고 가정해보자.   
이 경우 과금 정도를 측정하기 위해선 로그를 몇개 누적했는지를 추적해야 한다.   
하지만 count 자체는 비용이 큰 연산이니, Hyperloglog를 사용해서 count 전용 set을 만들어 여기에 로그를 누적할때마다 로그의 고유값(해시)를 넣어준다고 가정한다면 과금 정도를 간단하게 측정할 수 있다.
